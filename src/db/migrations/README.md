# Migrations

Mixed authorship: `0000`–`0015` were generated by `drizzle-kit generate` (each has a matching
`meta/NNNN_snapshot.json`). `0016`–`0021` are **hand-written** SQL (cascade FKs, enum rebuilds,
trigram indexes, the dual-axis taxonomy) — things drizzle-kit can't express 1:1.

## Why only the latest hand-written migration has a snapshot

`drizzle-kit generate` computes a diff against the **latest snapshot that exists on disk**, not
against every journal entry. So to keep `bun run db:generate` drift-free we only need the newest
migration to carry a current-state snapshot. That is `meta/0021_snapshot.json` — it equals the live
schema, so `generate` reports *"No schema changes"* instead of re-proposing the content_type enum
rebuild and the other hand-written DDL as phantom drift.

`0016`–`0020` intentionally have **no** snapshot. They don't need one: runtime `db:migrate` ignores
snapshots entirely (it tracks applied migrations in the `__drizzle_migrations` table by tag), and
`generate` only reads the latest. Do **not** hand-fabricate intermediate snapshots to "fill the
gap" — it adds risk for zero benefit.

## Adding the next migration

- **Schema-expressible change:** add the column/index to `src/db/schema.ts`, run
  `bun run db:generate`. Because `0021_snapshot.json` is current, the new migration + snapshot will
  be a clean incremental diff.
- **Hand-written DDL** (enum rebuild, `USING` retype, `CREATE EXTENSION`, data backfill): write the
  `NNNN_*.sql` by hand, append an entry to `meta/_journal.json`, and — if you want `generate` to
  stay drift-free afterward — regenerate the latest snapshot (generate once, keep the snapshot,
  discard the redundant SQL, as was done for `0021`).
