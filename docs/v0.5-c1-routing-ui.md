# v0.5 切片四 · C1 模型路由 UI（可编辑） — 设计

> 上游：[v0.5-diy-plan.md](./v0.5-diy-plan.md) C1。让主理人在 /_admin 里改每个 LLM 任务的 provider/model。
> 现状：路由是**静态配置 + env**（`src/llm/routing.ts` 的 `llmRouting` + `LLM_*` env），`resolveProvider` **同步**、在判定热路径上。可编辑 ⇒ 需 DB 覆盖 + 管线读覆盖，是热路径改动，故小心。

## 锁定的设计决策

| # | 决策 | 理由 |
|---|---|---|
| **C1-1** | DB 覆盖表 `llm_routing_overrides(task PK, provider, model)`；**只覆盖 provider+model** | promptVersion/maxTokens/temperature 与 prompt 逻辑绑定、不该用户调；mockup 也只给 provider/model。 |
| **C1-2** | **worker 侧内存缓存 + 1 分钟 cron 刷新**；`resolveProvider` 仍**同步读缓存** | 不能每次判定查 DB（热路径）；不能把 resolveProvider 改异步（牵连所有调用点）。缓存 + 定时刷新 = 热路径零改动，UI 改动 ≤1min 生效。web 进程改 DB，worker 进程刷自己的缓存。 |
| **C1-3** | 优先级：**DB 覆盖 > env > 默认** | UI 是「实时控制」，覆盖最高优先。无覆盖时回退既有 env/默认解析（不动 `llmRouting` 的 getter 链）。 |
| **C1-4** | **fail-closed 保持不变** | `getRouteConfig` 合并覆盖后，`resolveProvider` 照旧 `providerConfigured(provider)`：覆盖到没配 key 的 provider ⇒ 该任务 fail-closed（judge_failed no_key）。UI **显示每 provider 的 key 状态**，避免误配。 |
| **C1-5** | 覆盖值在**写入时校验**（合法 LlmTask + LlmProviderName）；`getRouteConfig` 再校验一次，非法覆盖忽略回退 base | 防止脏缓存把热路径打挂。 |

## 数据流
```
[web /_admin/routing]  POST /api/_admin/routing (admin gate + Zod + audit)
        │ upsert/delete
        ▼
  llm_routing_overrides (DB)
        │  每分钟 cron: refresh-routing-overrides
        ▼
[worker] loadRoutingOverrides() → setRoutingOverrides(cache)   (boot 时也 load 一次)
        │  resolveProvider(task) → getRouteConfig(task) = llmRouting[task] ⊕ cache.get(task)
        ▼
  判定管线（同步、热路径零改动）
```

## 闸门
每刀 tsc + `bun test src worker` + 集成（`--env-file=.env.nonexistent`）+ build 全绿独立提交。攒进 v0.5.0。

## 风险
- web/worker 进程分离：UI 改动靠 worker cron 同步，≤1min 延迟（路由变更非实时，可接受）。
- 覆盖到无 key 的 provider 会让该任务 fail-closed —— 由 UI 的 key 状态列提示，避免误操作。
- 缓存在 worker 内存：worker 重启会空一瞬（回退静态/ env，安全），boot 初次 load 补上。
