## 1. 背景与目标

AIWatch 是策展式 AI-Dev 情报阅读器（Bun + Next.js 15 + Postgres/Drizzle + graphile-worker）。本轮目标是**把"总结摘要能力"做强**，并把分类法理顺。明确决策（均由用户逐条拍板）：

1. **定位：纯 AI-Dev 垂直**——不扩泛科技。BestBlogs 覆盖编程/AI/产品/科技，但我们只学它的**摘要手艺**（六维打分 + 摘要 + 关键观点的结构化做法），不学它的覆盖面，也不学它的个性化（个性化本轮不做）。
2. **摘要分层由打分驱动**：只有精选（`score≥80`）才有资格详细输出；非精选**极致硬卡**。
3. **分类法：双轴**（领域 × 内容类型），取 aihot 5 类之神、去其"混轴"之弊。
4. **不逐句引用**（无金句），版权安全。

### 非目标（YAGNI）

- 不做个性化 / 推荐画像（BestBlogs 的 Pro 能力不学）。
- 不扩泛科技。
- 不引入逐句金句（`gold_quote`）。
- 不把现有 5 评分维度强行扩成 BestBlogs 的 6 维（避免动评分契约、避免撞上进行中的迁移）。

## 2. 分类契约：双轴（取代旧 §1 单轴）

**两个正交的轴**，每条事件各取其一（领域多选留待后续；MVP 单选）。

### 2.1 领域轴（主题；驱动浏览 Tab）

| 领域 | 收容 |
|---|---|
| 大模型 | 模型发布/权重/榜单、能力、架构、训练、scaling |
| 框架·工具 | 框架/SDK/库、Agent 框架、API、开发工具、工程底层 |
| 产品·应用 | AI 产品与功能、落地应用、商业化迭代 |
| 行业·商业 | 融资/收购/政策、生态级行业动态 |
| 科研·论文 | 论文、研究结论、方法、技术方向判断 |
| 安全·对齐 | 安全、对齐、红队、滥用、治理 |

> **开放问题（评审需拍）**：`科研·论文`（领域）与下方 `研究 paper`（类型）语义重叠——一篇"某模型的论文"该落 `大模型×研究` 还是 `科研·论文`？建议二选一：(a) 领域 `科研·论文` 仅收**不绑定具体产品/模型的纯研究机构产出**，其余论文走 `大模型/框架×研究`；或 (b) 把领域改名为纯主题（如 `基础设施`/`多模态`），论文性质完全交给类型轴。我倾向 (a)，保留用户已认可的 6 个领域名。

### 2.2 内容类型轴（形态；驱动事件折叠 + 摘要形态）

`发布 release` / `研究 paper` / `教程实操 howto` / `观点讨论 opinion` / `行业动态 news`

- 复用现有 `action_class` 机制做**事件折叠键**（`fold_key = normalize(primary_entity) + "|" + content_type`）。
- 类型也决定摘要骨架（如 `发布` 重"变了什么/影响"，`研究` 重"方法/结论"，`教程` 重"可照做的步骤"）。

### 2.3 Trash

不属于以上、或非 AI-Dev 的一律 `Trash` 丢弃（沿用旧 §1）。`$0` 关键词闸保留（垂直定位下无需重写）。

## 3. 评分与分层输出（沿用旧 §2，收紧硬卡）

| score | 档 | 处置 | 生成内容 |
|---|---|---|---|
| `<60` | T0 | 丢弃，不入库 | 无 |
| `60–79` | T1（非精选） | 极致硬卡 | **仅** `标题` + `一句 40–60 字结论`。要点/标签/详情一律不生成、不展示 |
| `≥80` | T2（精选） | 详细 | 在 T1 基础上：`详细摘要(100–150字)` + `2–3 条关键观点` + `tags` |

- **详细资格在入库判分时即定**（`aiScore≥80`），**不**等赞踩/promotion——因为深抽取发生在入库，运营信号尚不存在。
- **术语澄清**：此处"精选 = T2（score 门）"，与读者侧运营层的 `selectedLevel`(B/A/S) / `mode=selected` 是**两件事**，实现时勿混用同一个"精选"字眼。

## 4. 摘要流水线：单模型·按需两次

同一个模型承担两类调用（不分 cheap/strong 两级——这是用户对"不做两重"的定义）：

```
存活项 ──(调用①: triage)──> score + 领域 + 类型 + 一句话(40–60字)
                               │  代码卡门 (旧 §3 gate)
              score<60 丢弃 ───┤
              60–79 T1 ────────┤ 到此为止（一句话即成品）
              ≥80  T2 ──(调用②: 深抽取，单次)──> 详细摘要 + 关键观点 + tags
```

- 调用② **单次生成、不二次精炼**（"一重"）。
- 折叠（旧 §4）发生在调用①之后、调用②之前：命中已有事件则 append 来源、不重复深抽。
- **成本特征**：triage 对全量跑（单模型价，非 cheap 价），深抽只对 T2 跑。比旧"单次一遍过"省（非精选不付详细 token），比"两级模型"略贵（triage 用同一模型）。用户已接受此权衡。
- **模型选型**：留给 plan——需在"triage 跑全量够便宜"与"深抽够强"之间取一个中档、中文摘要好的模型。

### 4.1 Prompt 契约（受版本管控；扩展现有 `prompts.ts`）

- **triage prompt**（全量）：输出严格 JSON `{ score, domain, content_type, one_line_summary, fold:{primary_entity} }`。打分标准沿用旧 §2（90-100 重磅 / 80-89 干货 / 60-79 常规 / <60 Trash）；强调"无链接/无热词不影响打分"。
- **deep prompt**（仅 T2）：输出 `{ detailed_summary, core_viewpoints[2-3], tags[3] }`。结构"背景 → 发生了什么 → 对开发者/研究者的影响"。**显式禁止逐句引用**。
- 两个 prompt 各自带 version 常量；改动须过 Golden Set（旧 §7）才可合并。

## 5. 数据模型影响

- `events` / `event_judgments`：新增 `domain`（领域轴 enum）。`content_type` **复用现有列**，但取值切换到 §2.2 的 5 类。这是一次**非一一对应**的重映射——旧 4 值（`model_release` / `product_release` / `tech_share` / `discussion`）与新 5 值（`发布` / `研究` / `教程实操` / `观点讨论` / `行业动态`）数量不等：旧的两个 `*_release` 多半折叠进 `发布`，而新值 `研究`、`行业动态` 在旧表里无直接来源，回填时须按标题/来源二次判定。精确映射表 + 回填脚本是 plan 阶段交付物（见 §9.4）。
- `fold_key` / `simhash`：随进行中的迁移落地（见 §7）。
- **金句**：确认 schema 内**无** `gold_quote` 字段（与进行中迁移的 `deepExtractSchema` 一致）。

## 6. 兜底 / 质量闭环（沿用旧 §6/§7）

- 解析失败/缺字段 → 标 `judge_failed`，不写脏数据（已实现）。
- spend_guard fail-closed（已实现）。
- **Golden Set 是合并闸**：含硬负例（像干货的软文→Trash）、安静正例（纯文字洞察→高分）、各类目边界样本。triage/deep 两 prompt 任一改动必须过闸。

## 7. 与"进行中迁移"的关系（重要）

审计发现工作树有一个正在实时进行的"pipeline 契约迁移"（新增 `prompts.ts`/`simhash.ts`/`golden-set.test.ts`/迁移 `0018_pipeline_contract.sql`，重写 `judge-schema.ts`/`process-source.ts`）。本设计与它**两处分叉**，收尾者需对齐：

1. **模型分级**：迁移在做 cheap(light) + strong(deep) **两级模型**；本设计改为**单模型两次调用**。→ 路由配置（`llm/routing`）应退化为单模型，light/deep 仅为两个 prompt，不为两个 provider 级别。
2. **分类法**：迁移用旧 §1 的 4 个 `INTELLIGENCE_CATEGORIES`（Core_Research…）；本设计改为**双轴（领域6 × 类型5）**。→ `judge-schema` 的 category 契约、`0018` 迁移、reader 筛选 facet 三处都要按双轴改。
3. **对齐项**：无金句、score→T0/T1/T2 门控、SimHash 折叠、Golden Set——本设计与迁移一致，直接复用。

> 当前工作树不通过 typecheck（迁移半截）。**先让迁移落地并收绿**（用户已约定届时做"运行期真验证"），再在其上实施本设计的分叉调整，避免双写冲突。

## 8. 验收

- triage/deep 两 prompt 过 Golden Set；双轴分类在边界样本上稳定。
- T1 项确认只产出 标题+一句；T2 项产出详细摘要+2-3 观点+tags；无逐句引用。
- typecheck / 单测 / 集成 / `next build` 全绿；reader 双轴筛选可用。
