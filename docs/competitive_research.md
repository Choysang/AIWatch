# 竞品研究（只抽象机制，不复制实现）

研究目的：理解"AI 信息聚合/严选"品类的常见能力与产品机制，校准 AIWatch 的差异化路线。
本文所有结论都是机制层面的抽象总结；UI、文案、信源池、代码一律自研。

## 1. BestBlogs（RSS 严选 + AI 分析流水线）

可借鉴的机制：
- **六阶段流水线**：抓取 → 初筛 → AI 深度分析 → 翻译 → 入库 → 推荐。与我们"轻判 → 深抽"两段式同构；它的"初筛挡住低价值再做贵的深度分析"与我们的 T1/T2 分流一致。
- **多维评分 + 显式加减分**：内容深度/相关性/写作质量/实用性主维度之外，对"营销味、篇幅过短、泛泛而谈"显式扣分。我们已把"严格扣分"条款写进轻判提示词（v4）。
- **反思自检**：分析产出后再做一轮"全面性/准确性/一致性"检查。我们以轻量方式纳入深抽提示词的自检条款，而非独立 LLM 调用（控制成本）。
- **术语表约束翻译**：预建领域术语映射保证中文一致性。已落地为 `data/glossary/ai_terms_zh.json` + 深抽提示词术语规范。
- **每日早报/精选周刊的"二次提纯"**：日报不是流水账，而是再筛选。我们的日/周/月报同理（确定性生成）。
- **OpenAPI/Agent 接入**：内容池暴露为可编程接口。我们已有公共 API + RSS + MCP/Skill。

差异化坚持：BestBlogs 的评分由 LLM 直接给总分；我们坚持 **LLM 只产不可变结构化输入，总分由确定性代码/SQL 合成**——调权重等于重跑，不是重新推理。

## 2. AIWatch（时间线式 AI 动态雷达）

可借鉴的机制：
- **多信源热度**：同一事件被多个信源报道时聚合为一条并展示信源数，热度随时间消退。我们的 fold_key 事件折叠 + 当前热点模块同向，可进一步把"信源数"作为热度可视信号（已记入 future_ideas）。
- **分类即导航**：精选/全部/模型/产品/行业/论文/技巧作为一级入口。我们已有 精选/全部 + 四分类（产品/技术/技巧/讨论）+ 内容类型内部轴。
- **推荐理由层**：摘要之下补一行"为什么值得看"。我们卡片已有推荐理由字段（精选项展示）。
- **信源提报与反馈入口**：社区补充信源。已有（推荐信源 + 意见反馈 + 后台审核）。
- **日报按日分组、精确时间戳**：已有（日报 + 时间线日分组）。

## 3. 其他同类项目（机制扫描）

| 项目 | 类型 | 可借鉴机制 |
|---|---|---|
| NewsNow（开源热榜聚合） | 多源热榜 | 极轻部署（一进程 + SQLite/缓存）、榜单"最后刷新时间"显著展示、源失效自动降级。我们已加"最后更新"提示；轻部署对齐（compose 三容器）。 |
| Folo / Follow（开源 AI 阅读器） | 订阅+AI | 订阅源健康可视化、AI 摘要按需生成（点开才花钱）。我们信源健康后台已有；"按需深抽"思路与 T2 门槛一致。 |
| wewe-rss（开源公众号桥接） | 微信→RSS | 自托管公众号 RSS 的可行路径。当前因合规/稳定性把公众号整体排除在核心池外；若未来要做，自托管 wewe-rss 优于依赖任何第三方桥接。 |
| Meridian（开源 AI 简报） | 情报简报 | "事件聚类 → 重要性排序 → 生成简报"的全自动链路；简报附引用源列表。我们日报同构，引用源已展示。 |
| Hacker News digest 类（多个开源实现） | 单源精选 | 用阈值（分数/评论数）做确定性初筛再 LLM 摘要——便宜的预过滤放在 LLM 之前。我们的门控/去重层同理。 |
| TrendFinder（开源社媒监控） | X 监控 | 用自托管 RSSHub/API 拉 X 内容 + LLM 判定是否值得推送；与我们 X 信源链路一致，验证了路线可行性。 |

## 4. 明确不采用

- 不复制任何竞品的 UI、视觉风格、文案、信源池、命名体系。
- 不做重用户系统（关注/画像/个性化推荐留到有真实用户行为之后）。
- 不做商业化功能（会员、付费墙、积分）。
- 不引入重依赖（Algolia/ES/Redis/消息队列等）：搜索用 pg_trgm，队列用 graphile-worker，缓存靠 SQL + Next 缓存。
- 不做播客/YouTube/TTS 链路：文本雷达主线优先，音视频需要转录管线，复杂度/成本不匹配当前阶段。
- 不做"大而全"信息流：宁缺毋滥，30 个严选源 + 严格扣分提示词。
